좋습니다.
그렇다면 이제 프로젝트의 정체성은 꽤 명확합니다.

**이 앱은 “AI가 주인공인 챗봇 앱”이 아니라, 플랫폼별 화제 주제를 수집·정리·비교해 보여주는 트렌드 수집형 서비스**이고, AI는 그 데이터를 해설하고 확장하는 보조 계층으로 두는 것이 맞습니다.
특히 현재 기준으로는 YouTube는 공식 API로 지역/카테고리 기준 인기 영상 차트를 가져오기 좋고, X는 트렌드/실시간 데이터 접근이 가능하지만 사용량 기반 과금 구조를 고려해야 하며, Instagram은 해시태그 검색이 가능하더라도 Business/Creator 계정 제약과 7일 기준 30개 고유 해시태그 제한이 있어 “플랫폼 전체 인기 주제 랭킹” 소스로 쓰기엔 불리합니다. Google Trends는 공식 API(alpha)가 발표되었지만 아직 일반적인 안정 채택 대상으로 보기엔 이른 편입니다. ([Google for Developers][1])

## 1) 지금 단계에서 고정해야 할 핵심 해석

여기서 가장 중요한 설계 결정은 이것입니다.

**“우리는 검색어 앱이 아니라, 플랫폼별 화제 주제 앱을 만든다.”**

이 말은 곧 내부 모델도 바뀐다는 뜻입니다.
즉, DB와 API를 `keyword` 중심으로만 잡지 말고, 더 상위 개념인 `topic` 중심으로 잡아야 합니다.

예를 들면:

- YouTube: 인기 영상 제목/설명/태그에서 추출한 화제 주제
- X: 트렌딩 토픽/해시태그/주제
- Instagram: 해시태그 기반 관심 주제
- Google Trends: 검색 관심도 기반 주제

즉, 화면에는 “주제”로 보이되, 내부에는 각 플랫폼 원본 데이터를 따로 저장하고, 그 위에 **정규화된 Topic 엔티티**를 얹는 구조가 좋습니다.

## 2) 놓치기 쉬운 설계 관점

### 관점 A. 원본 데이터와 정규화 데이터를 분리해야 합니다

이걸 안 하면 나중에 플랫폼이 추가될 때 구조가 무너집니다.

권장 분리:

- **RawTrendItem**: 플랫폼 응답 원본
- **Topic**: 정규화된 화제 주제
- **TopicSnapshot**: 날짜별/플랫폼별 순위 기록
- **TopicAnalysis**: AI 분석 결과
- **UserInterest**: 사용자 관심 분야
- **Recommendation**: 추천 결과
- **Post/Comment**: 커뮤니티

핵심은 `Topic`과 `TopicSnapshot`을 분리하는 것입니다.
그래야 “어제 8위였는데 오늘 3위로 올랐다” 같은 흐름이 가능합니다.

### 관점 B. AI는 생성보다 해설에 집중해야 합니다

학부생 2인 프로젝트에서 AI가 추천 논리까지 전부 담당하게 만들면, 품질 제어가 너무 어려워집니다.

그래서 AI의 책임은 이렇게 제한하는 게 좋습니다.

- 유행 이유 요약
- 관련 키워드 추출
- 주제 설명 생성
- 간단한 추천 문장 생성

반대로 아래는 규칙 기반이 더 좋습니다.

- 순위 계산
- 카테고리 분류
- 사용자 관심사 매칭
- 저장한 주제와 유사한 주제 추천

즉, **판단의 뼈대는 서버 로직, 표현은 AI**로 두는 게 안정적입니다.

## 5) 플랫폼별 전략은 이렇게 나누는 게 좋습니다

### 1. YouTube

가장 안정적인 1순위 데이터 소스입니다.
`videos.list`의 `chart=mostPopular`로 지역별 인기 영상을 조회할 수 있고, `regionCode`, `videoCategoryId`를 함께 사용해 카테고리별 차트도 구성할 수 있습니다. 다만 2025년 7월 이후 `mostPopular`의 의미가 예전 “Trending Now”와 같지 않고, 음악·영화·게임 등 차트 기반으로 조정된 점은 설계 시 감안해야 합니다. ([Google for Developers][1])

### 2. X

트렌드 데이터와 실시간성 측면에서는 매력적입니다.
다만 현재는 사용량 기반 과금과 엔드포인트별 비용 구조를 고려해야 하므로, 졸업작품 범위에서는 호출 빈도를 낮추고 저장 위주 구조로 가는 것이 안전합니다. X 공식 문서상 트렌드 접근과 위치 기반 트렌드 사용이 가능하다는 점은 장점입니다. ([X Developer Platform][2])

### 3. Instagram

“인기 주제 전체 랭킹”보다는 해시태그 중심 접근이 맞습니다.
공식 문서 기준 해시태그 검색은 Instagram API with Facebook Login/Graph API 흐름과 계정 제약이 있고, 7일 동안 30개 고유 해시태그 제한도 있어 대규모 수집 소스로는 비효율적입니다. 따라서 초기에 Instagram은 메인 소스보다 **보조 소스**로 두는 것이 맞습니다. ([Facebook Developers][3])

### 4. Google Trends

공식 API(alpha)가 발표된 것은 긍정적이지만, 아직 일반적인 안정 서비스 의존 대상으로 보기엔 조심스러운 단계입니다. 그래서 발표/구현 일정이 빡빡하다면 “향후 확장 가능 소스” 정도로 두는 것이 현실적입니다. ([Google for Developers][4])

## 6) 설계 대안 2가지 비교

### 대안 A. 플랫폼 2개를 깊게

- YouTube + X
- 주제 조회/상세/분석/게시판까지 완성

장점:

- 구현 안정성 높음
- 데이터 흐름 설명이 명확
- 발표 때 실패 확률 낮음

단점:

- 플랫폼 수가 적어 보여서 겉보기 확장성은 약함

### 대안 B. 플랫폼 4개를 얕게

- YouTube + X + Instagram + Google Trends
- 각 플랫폼은 일부만 연동

장점:

- 겉으로 보기에는 풍성함
- “다양한 플랫폼 지원” 강조 가능

단점:

- API 제약 대응 비용 큼
- 오류/예외 처리 폭증
- 2인 프로젝트에서는 유지보수 부담 큼

### 제 추천

무조건 **대안 A**입니다.

이 프로젝트는 데이터 수집 능력이 주가 되므로,
“지원 플랫폼 수”보다 **“수집 → 저장 → 분석 → 비교 → 화면 반영”의 완성도**가 더 중요합니다.

즉:

- 1차 시연: **YouTube + X**
- 2차 확장: Google Trends
- 선택 확장: Instagram

이 순서가 가장 건강합니다.

## 8) API도 처음부터 화면 기준으로 설계하는 게 좋습니다

### 홈

- `GET /api/home`
  - 오늘의 주요 주제
  - 플랫폼별 top N
  - 추천 주제 일부
  - 인기 게시글 일부

### 플랫폼별 목록

- `GET /api/trends?platform=youtube&date=today`
- `GET /api/trends?platform=x&date=2026-03-20`

### 주제 상세

- `GET /api/topics/{id}`
  - 기본 정보
  - 최근 순위 변화
  - 관련 키워드
  - AI 분석
  - 관련 게시글

### 분석

- `POST /api/topics/{id}/analyze`
  - 최초 분석 생성

- `GET /api/topics/{id}/analysis`
  - 저장된 분석 조회

### 추천

- `GET /api/recommendations/me`

### 커뮤니티

- `GET /api/posts`
- `POST /api/posts`
- `GET /api/posts/{id}`
- `PUT /api/posts/{id}`
- `DELETE /api/posts/{id}`
- `POST /api/posts/{id}/comments`

이렇게 하면 프론트도 화면 단위로 작업하기 편합니다.

## 10) 유지보수성과 확장성 관점 검증

이 구조가 좋은 이유는 다음과 같습니다.

### 확장성

플랫폼이 하나 더 추가돼도 `collector`만 늘어나고 `topic` 중심 구조는 유지됩니다.

### 유지보수성

AI 분석이 실패해도 트렌드 조회 기능은 유지됩니다.

### 테스트 용이성

- 수집기 테스트
- 정규화 테스트
- 추천 규칙 테스트
- 게시판 CRUD 테스트
  를 분리할 수 있습니다.

### 발표 안정성

API가 일부 실패해도 미리 저장된 snapshot으로 시연이 가능합니다.

즉, **실시간 의존도를 줄이고 저장 기반 앱으로 설계하는 것**이 졸업작품에는 특히 중요합니다.

## 12) 결론

지금 잡으신 방향은 꽤 좋습니다.

정리하면:

- **핵심 정체성**: 플랫폼별 화제 주제 수집 앱
- **핵심 강점**: 데이터 수집 및 비교
- **AI의 역할**: 해설, 요약, 관련 키워드 확장
- **추천 스택**: React Native + Django REST Framework + MySQL + Gemma API
- **추천 전략**: 플랫폼 2개를 깊게, 나머지는 확장 후보로
- **핵심 내부 모델**: Raw 데이터와 Topic/Snapshot 분리

그리고 설계적으로 가장 중요한 한 줄은 이것입니다.

**“플랫폼 데이터를 그대로 보여주는 앱”이 아니라, 플랫폼 데이터를 정규화해서 ‘주제’로 재구성하는 앱”**으로 가야 합니다.

다음 단계로 이어가려면, 저는 이제
**화면별 스토리보드 구조표 + API 명세 초안 + DB 테이블 초안**까지 한 번에 묶어서 설계해보는 게 가장 좋다고 봅니다.
