## 2026.03.19. chat

이 프로젝트는 기능을 많이 넣는 것보다 **“2명이서 학부생 수준으로 끝까지 구현·시연 가능한 구조”**로 다시 잘 쪼개는 게 가장 중요합니다.

핵심부터 말하면, 이 앱은 사실상 하나의 앱이 아니라 아래 **3개의 서브시스템**으로 나뉩니다.

1. **트렌드 수집 시스템**
   각 플랫폼에서 인기 키워드/인기 콘텐츠를 가져와 저장

2. **AI 분석·추천 시스템**
   키워드 의미 분석, 유행 이유 요약, 관련 키워드 확장, 사용자 맞춤 추천

3. **커뮤니티 시스템**
   관심 트렌드 기반 게시판, 댓글, 저장, 간단한 상호작용

그리고 여기서 가장 중요한 현실 조건은 **데이터 소스별 구현 가능성 차이**입니다.
YouTube는 공식 API로 인기 영상 차트 조회가 비교적 명확하고, `mostPopular` 차트를 지역/카테고리 기준으로 가져올 수 있습니다. 반면 Google Trends는 2025년 7월에 공식 Trends API(alpha)가 발표됐지만 제한된 테스터에게만 제공되고, X는 현재 사용량 기반 과금 구조의 API를 제공하며, Instagram은 일반적인 “실시간 인기 검색어 순위”보다는 해시태그 검색 및 관련 미디어 조회 중심입니다. 그래서 **초기 졸업작품 범위에서는 YouTube + Google Trends(가능 시) + X(가능 시) + Instagram(옵션)**처럼 우선순위를 나누는 것이 안전합니다. ([Google for Developers][1])

## 1) 문제의 본질부터 구조화해보면

이 앱이 사용자에게 주는 가치는 단순히 “인기 검색어 보여주기”가 아닙니다.

실제로는 이런 흐름입니다.

**데이터 수집 → 정규화 → 저장 → 분석 → 추천 → 사용자 참여**

즉, 사용자 입장에서 보이는 화면은 예쁘게 4~5개 기능으로 보이지만, 개발 관점에서는 다음 질문에 답해야 합니다.

- 어떤 플랫폼의 어떤 데이터를 가져올 것인가
- 서로 다른 플랫폼 데이터를 어떤 공통 형식으로 저장할 것인가
- “인기”를 무엇으로 정의할 것인가
- AI는 어디까지 자동화할 것인가
- ~~커뮤니티는 단순 게시판인지,~~ 트렌드와 연결된 게시판인지

이 질문을 먼저 정리하지 않으면 스토리보드가 예뻐도 구현 단계에서 무너집니다.

## 2) 놓치기 쉬운 설계 관점

이 프로젝트에서 가장 많이 놓치는 부분은 **“검색어”와 “플랫폼 원본 데이터”를 분리하지 않는 것**입니다.

예를 들어 YouTube는 엄밀히 말하면 “검색어 순위”가 아니라 **인기 영상 차트**입니다.
Instagram도 일반 검색어 랭킹보다는 **해시태그·콘텐츠 중심**으로 접근하게 될 가능성이 높습니다.
X 역시 “플랫폼 내 화제성 데이터”를 어떤 방식으로 추출하느냐가 관건입니다. ([Google for Developers][2])

그래서 내부 모델은 이렇게 분리하는 게 좋습니다.

- **SourceRawData**: 플랫폼 원본 응답
- **TrendKeyword**: 정규화된 키워드
- **TrendSnapshot**: 날짜별/플랫폼별 순위 기록
- **TrendAnalysis**: AI 분석 결과
- **UserInterest / Recommendation**: 사용자 관심사 및 추천 결과
- **CommunityPost / Comment**: 게시판 데이터

이렇게 해야 나중에 데이터 소스가 바뀌어도 앱 전체를 갈아엎지 않습니다.

### B. 검색어 기반 사용자 맞춤 AI 추천

이 기능은 “멋있어 보이지만 범위 폭발”이 가장 쉽게 나는 부분입니다.
그래서 추천을 3단계로 단순화하는 게 좋습니다.

#### 추천 방식 1: 관심사 기반

사용자가 관심 카테고리를 고름
예: 패션 / 게임 / K-POP / IT

→ 해당 카테고리의 최근 인기 키워드 추천

#### 추천 방식 2: 저장 키워드 기반

사용자가 저장한 키워드와 유사한 키워드 추천
예: “러닝화”, “고프코어” 저장
→ “트레일러닝”, “아웃도어룩”, “러닝웨어” 추천

#### 추천 방식 3: 행동 기반

사용자가 자주 클릭한 키워드나 읽은 게시글을 기반으로 추천

#### 추천 결과 화면 예시

- 당신이 관심 가질 만한 트렌드
- 비슷한 키워드
- 요즘 함께 뜨는 키워드
- 관련 게시글 보기
- AI 한줄 추천 이유

#### 학부생 수준 추천 구현 방식

초기에는 복잡한 추천 알고리즘보다

- 카테고리 매칭
- 키워드 유사도
- 태그 기반 룰 추천
- AI 요약 문장 생성

이 정도로도 충분합니다.

즉, 추천 엔진은 처음부터 ML 모델이 아니라
**“규칙 기반 + LLM 문장화”**로 가는 게 맞습니다.

### C. 트렌드 단어 분석 기능

이 기능이 이 앱의 차별점이 될 가능성이 큽니다.
단순 순위 앱은 많지만, **“왜 유행하는지 설명해주는 기능”**은 시연 포인트가 좋습니다.

#### 분석 항목 추천

- 이 단어가 무엇인지
- 왜 유행하는지
- 어떤 플랫폼에서 강한지
- 관련 키워드
- 유사 트렌드
- 긍정/부정/중립 분위기
- 예상 지속성
  - 단기 밈
  - 중기 유행
  - 장기 관심사

#### AI 분석 결과 예시

키워드: “고프코어”

- 정의: 아웃도어 기능성 스타일을 일상 패션에 결합한 트렌드
- 유행 이유: 실용성, 레트로 감성, SNS 착장 공유
- 관련 키워드: 아크테릭스, 살로몬, 러닝룩
- 연관 카테고리: 패션, 라이프스타일, 운동
- AI 해석: 실용성과 스타일을 동시에 추구하는 소비 패턴과 연결됨

#### 구현 방식

- 수집된 키워드를 DB에 저장
- 키워드 상세 요청 시 Gemma API 호출
- 응답 결과를 캐시/DB 저장
- 같은 키워드는 재요청 대신 저장된 분석 결과 사용

이게 중요한 이유는 **API 비용 절감**과 **응답 속도 개선** 때문입니다.

### E. AI 챗봇 (후순위)

이건 진짜 후순위가 맞습니다.

이유는 챗봇 하나가 사실상 별도 프로젝트이기 때문입니다.

#### 챗봇이 할 수 있는 일

- “요즘 20대 패션 트렌드 알려줘”
- “어제보다 많이 오른 키워드 뭐야?”
- “고프코어랑 비슷한 트렌드 추천해줘”
- “이 키워드 관련 게시글 보여줘”

#### RAG 구성 최소 버전

- 내부 DB의 트렌드 기록
- 분석 결과 캐시
- 게시판 글 일부
- 이 데이터만 검색해서 답변

#### 현실적 판단

졸업작품에서 챗봇은

- 발표용 데모 포인트는 좋지만
- 개발 난이도와 안정성 관리가 큼

그래서 **핵심 기능 완성 후 추가**가 맞습니다.

## 4) 설계 대안 2가지 비교

### 대안 A. “데이터 수집 중심형”

먼저 플랫폼별 트렌드 수집을 확실히 만들고, AI는 그 위에 얹는 방식

#### 장점

- 트렌드 앱의 본질에 충실
- 데이터가 쌓이면 시연이 안정적
- AI가 실패해도 기본 앱은 돌아감

#### 단점

- 초기에 화면이 다소 밋밋할 수 있음
- “AI 앱”처럼 보이는 임팩트는 약함

#### 적합한 경우

- 안정적인 졸업작품
- 발표 때 실제 데이터 흐름을 강조하고 싶을 때

---

### 대안 B. “AI 분석 중심형”

수집 범위를 줄이고, 키워드 분석/추천 화면을 더 화려하게 만드는 방식

#### 장점

- 시연 임팩트가 큼
- 차별화 포인트가 명확
- 사용자 입장에서 재미있음

#### 단점

- 원천 데이터가 빈약하면 신뢰도 하락
- LLM 응답 품질에 서비스 가치가 많이 의존

#### 적합한 경우

- 발표에서 “AI 해석 능력”을 강조하고 싶을 때
- 수집 소스가 제한적일 때

---

### 제 추천

당장 팀 규모와 졸업작품 현실성을 고려하면
**A 70% + B 30%**가 제일 좋습니다.

즉,

- 수집/조회 기능은 안정적으로
- AI 분석은 핵심 상세 화면에 집중적으로
- 챗봇은 후순위

이 구조가 가장 안전합니다.

## 5) Flask vs Django 비교

### Flask 선택 시

#### 장점

- 가볍고 구조를 자유롭게 설계 가능
- API 서버 중심으로 만들기 좋음
- 학습 목적상 내부 구조를 직접 설계하기 좋음

#### 단점

- 인증, 관리자 페이지, 게시판, ORM, 권한 관리 등을 직접 더 많이 붙여야 함
- 2인 프로젝트에서는 생산성이 떨어질 수 있음

---

### Django 선택 시

#### 장점

- 사용자 인증, ORM, 관리자 페이지, CRUD, 게시판 구현 속도가 빠름
- 커뮤니티 기능과 관리자 페이지 구성에 강함
- 졸업작품에서 결과물을 빠르게 안정화하기 좋음

#### 단점

- 구조가 다소 무겁게 느껴질 수 있음
- Flask보다 자유도는 낮음

---

### 제 추천

이번 프로젝트는 **Django 쪽이 더 적합**합니다.

이유는 명확합니다.

- 회원가입/로그인 필요 가능성
- 게시판 CRUD 필요
- 관리자 페이지 유용
- AI 분석 결과/트렌드 데이터 관리 필요
- 2명이서 빠르게 기능을 붙여야 함

즉, 이 프로젝트는 “가벼운 API 서버”보다
**“콘텐츠/유저/관리 기능이 있는 서비스”**에 가깝습니다.

React Native 프론트와는 Django REST Framework로 API를 주고받으면 충분합니다.

- created_at

### Comment

- id

- user_id

이 구조의 핵심은
**TrendKeyword와 TrendSnapshot을 분리**하는 것입니다.

## 12) 결론

이 프로젝트를 잘 만들려면 기능을 더 추가하는 것보다, 먼저 이렇게 정의해야 합니다.

**“우리 서비스의 핵심은 인기 데이터 수집 앱인가, AI 해설형 트렌드 앱인가?”**

제 판단으로는 가장 좋은 방향은:

**트렌드 조회를 뼈대로 두고, AI 분석을 차별화 포인트로 올리고, 커뮤니티를 서비스 완성도로 붙이는 구조**입니다.

그리고 기술 스택은 현재 조건이라면

**React Native + Django REST Framework + DB + Gemma API**
가 가장 안정적입니다.

마지막으로 설계 검증 질문 두 가지를 던져보겠습니다.

1. 여러분 팀이 말하는 “트렌드”는 **검색어 중심**인가요, 아니면 **플랫폼별 화제 주제 중심**인가요?
2. 발표에서 가장 보여주고 싶은 건 **데이터 수집 능력**인가요, **AI 해석 능력**인가요?

이 두 질문에 대한 답이 정리되면, 스토리보드와 API 명세 방향이 거의 확정됩니다.

원하시면 다음 단계로 바로 이어서
**“졸업작품용 기능 명세서 초안”** 또는 **“화면별 스토리보드 구조표”** 형태로 정리해드리겠습니다.
